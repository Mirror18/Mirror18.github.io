多线程

关于多线程的话，实话说还是要过一遍基础，因为忘的也差不多了。

首先就是概念上，是多任务，这个是跟CPU内核的分配有关，一般情况下是时刻分配任务。不过近代常见系统是抢占式多任务。

那么对于单核CPU也可以执行多任务，任务是什么，通过人的概念转换成计算机的概念，即任务=进程。但同时任务也可以分为微任务，即任务执行步骤，那么在计算机中就转换为线程。一个任务包含多个小任务，那么一个进程包含多个小线程，即线程。

于是再回到最现实中，CPU执行的是什么，是进程还是线程，嗯，这个有点麻烦，但是依旧可以看出来，进程包含执行资源，但是CPU切换的时候是切换线程。

补充点知识，即任务，因为任务是可以只有一个单任务，也可以是一个任务有多个执行步骤。这些定义很烦，总是会造成搞不清楚，这是与任务的定义有关，毕竟一个执行步骤也可以说是一个单任务。这是造成迷茫的问题所在。

但现在只考虑Java是怎么处理的，Java提供多线程支持，其实多进程也可以，就是启动project中的多个module。怎么区分的，区分点就在于是进程是包含任务资源的，例如申请的栈空间，所以栈切换很麻烦，要从寄存器中的资源做持久化，在CPU切换的时候需要改动。但是线程是没有内存资源的，访问的是进程所以同的资源。这也是说为什么线程切换速度快，但是不稳定，崩溃会影响其他线程。

那么就要考虑Java中多线程应用场景，读写网络，数据库，web开发，等都是多线程，都有一个特点，那就是需要等待，不管是内存也好，网络资源也罢，都是需要等待对方的回应才能继续下一步，或者说是需要对方提供资源才能进行下一步。为什么会这么说呢，继续往下看。

创建线程

// 多线程
public class Main {
    public static void main(String[] args) {
        Thread t = new MyThread();
        t.start(); // 启动新线程
    }
}

class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("start new thread!");
    }
}

// 多线程
public class Main {
    public static void main(String[] args) {
        Thread t = new Thread(new MyRunnable());
        t.start(); // 启动新线程
    }
}

class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("start new thread!");
    }
}

// 多线程
public class Main {
    public static void main(String[] args) {
        Thread t = new Thread(() -> {
            System.out.println("start new thread!");
        });
        t.start(); // 启动新线程
    }
}

方法很简单，虽然是三种方法，但其实都在一个核心点，那就是t.start()启动。启动的方法是run()，但是这个方法所在的类必须是Runnable。

另外要吐槽的就是，为什么要把这个打印语句的任务去交给其他方法去执行，为什么不直接在main方法中直接写出来，嗯，这也是回答上述问题，为什么要有多线程，说白了就是为了优化，因为有时候要执行一些比较耗时的操作，例如从磁盘中读取数据到内存中，比较耗时，CPU就可以在这段时间内执行其他任务，这就是多线程出现的意义，虽然我们代码是按照自己写的顺序执行的，但是但把一些耗时的操作放到其他线程中，这样可以做到有效利用。

那么多线程的执行顺序怎么确定呢，很好，不能确定，在没有启动线程前，是单独主线程执行，当启动线程后，就是两个线程并行，当然本质上是并发。只是在程序上这样理解，毕竟并行本质上是指多核CPU同时执行多个任务。并发就是在一段时间内一起执行。

以上是简单创建了一个多线程，那么肯定不能满足我们的日常需求，因为我们需要确定不同线程的执行顺序，例如主线程需要读取磁盘到数据，次线程是去读取磁盘，当还没有读取到的时候，就开始执行主线程，那么一定会出现错误，这种是可以解决的，只需要控制顺序即可，当然现在指的是异步哈，如果是串行是只是代码可视化上，比较好理解。

线程状态

一共有以下的状态

  • New：新创建的线程，尚未执行；
  • Runnable：运行中的线程，正在执行run()方法的Java代码；
  • Blocked：运行中的线程，因为某些操作被阻塞而挂起；
  • Waiting：运行中的线程，因为某些操作在等待中；
  • Timed Waiting：运行中的线程，因为执行sleep()方法正在计时等待；
  • Terminated：线程已终止，因为run()方法执行完毕。

第一个new 已经是在创建新线程的时候说了，在t.start()之前，然后Runnable是t,start()执行之后。

那么Blocked是指什么呢，就是请求的资源没有释放，但是当前CPU分片已经到达时间了，要切换线程，于是就选择挂起，当前还有其他的方式，主要就是运行中的线程，因为操作被阻塞而挂起。

waiting，运行中的线程因为操作而等待，举个简单的例子就是CPU分片还没到打时间，不需要切换线程，但是请求的资源还没有得到，于是在等待。

Timed Waiting 这个是指运行中的线程因为被执行Thread.sleep()而等待。

Terminated，指的是线程终止。就当前线程完成任务而结束了。

运行状态

线程获取到CPU的时间片。就会从就绪转到运行，但是当时间片用完或者调用yield()函数，就会切换到就绪状态

终止状态

线程继续运行，执行结束或者异常，就会进入到终止，这个操作不可逆，除非重新调用。

等待状态

这是指运行状态的线程执行wait()、join()、LockSupport.park()这几个函数中的一个，就会进入等待状态

其中wait()、join()函数是把线程放入到锁等待队列，不会被分配CPU执行时间，需要被主动唤醒。

notify()、notifyAll()、join线程执行完毕，会唤醒锁等待队列的线程，出队的现成回到就绪状态。

超时等待状态

这个跟等待状态的区别就是在于会有一个默认值，如果在值范围内的时间内，没有被唤醒，会自动唤醒。这些单位都是毫秒

  • wait(long)
  • join(long)
  • LockSupport.parkNanos(long)
  • LockSupport.parkUtil(long)
  • sleep(long)

中断线程

这个该怎么理解呢，就是多线程不管是在那个状态，都回到终止状态，通过调用

// 中断线程
public class Main {
    public static void main(String[] args) throws InterruptedException {
        Thread t = new MyThread();
        t.start();
        Thread.sleep(1); // 暂停1毫秒
        t.interrupt(); // 中断t线程
        t.join(); // 等待t线程结束
        System.out.println("end");
    }
}

class MyThread extends Thread {
    public void run() {
        int n = 0;
        while (! isInterrupted()) {
            n ++;
            System.out.println(n + " hello!");
        }
    }
}

调用的是interrupt()方法。但需要注意的是

不会立马中断，而是发出一个中断请求，如何处理这个中断请求还需要代码的处理，这种情况就是自身立刻结束运行。

那为什么会要设置一个中断请求呢，而不是直接结束，原因在于如果线程不处理这个中断请求的代码，那么其实释放中断是没有意义的。

原因如下

// 中断线程
public class Main {
    public static void main(String[] args) throws InterruptedException {
        Thread t = new MyThread();
        t.start();
        Thread.sleep(1000);
        t.interrupt(); // 中断t线程
        t.join(); // 等待t线程结束
        System.out.println("end");
    }
}

class MyThread extends Thread {
    public void run() {
        Thread hello = new HelloThread();
        hello.start(); // 启动hello线程
        try {
            hello.join(); // 等待hello线程结束
        } catch (InterruptedException e) {
            System.out.println("interrupted!");
        }
        hello.interrupt();
    }
}

class HelloThread extends Thread {
    public void run() {
        int n = 0;
        while (!isInterrupted()) {
            n++;
            System.out.println(n + " hello!");
            try {
                Thread.sleep(100);
            } catch (InterruptedException e) {
                break;
            }
        }
    }
}

通知t线程中断，但是t线程中没有中断信号的处理，只是处于等待状态，在上述我们说过，如果是使用join方法进入到等待状态，这时候是进入到锁的等待队列。

等待过程中获取到interrupt信号，会抛出异常。于是t线程继续执行。开始给Hello线程发出中断信号，这样我们才能中断Hello线程。那么这么做的原理是什么，换成人话讲就是说。

我只是发出一个信号量，然后线程内部去接收，至于接受不接受就看代码是如何写的。

那么这么说我们直接用一个布尔值来代替这种方法不就行了么，反正只是传递信号，自建一个不也行么。

// 中断线程
public class Main {
    public static void main(String[] args)  throws InterruptedException {
        HelloThread t = new HelloThread();
        t.start();
        Thread.sleep(1);
        t.running = false; // 标志位置为false
    }
}

class HelloThread extends Thread {
    public volatile boolean running = true;
    public void run() {
        int n = 0;
        while (running) {
            n ++;
            System.out.println(n + " hello!");
        }
        System.out.println("end!");
    }
}

对，这种方法没毛病，但是有个问题，那么就是join的时候怎么办，通过前面的论述，我们这个时候是处于等待队列的，我们是要等notify唤醒进入到就绪状态，然后再运行，之后才进入到终止状态。这也是不推荐用自己的方法来处理这种中断。因为无法保证线程现在处于的是什么状态。

同样的，我们看这个自建的标志位，会发现多了一个关键字，那就是volatile关键字，这是牵扯到线程之间的通信问题。

这个怎么说呢，原因就在于虚拟机上。我们程序读取变量是磁盘读取到内存。我们访问变量就是访问内存上的数据。但是有个问题是哪，就是每个类都有自己的初始化空间，一般来说常量都是保存在栈里面的，但是别的栈访问，一般的处理就是复制一份到另一个类中，在另一个类中做了更改，会在随机时间后写回。

这种随机事件写回只占用一个时刻，所以按照程序执行的时间来说，可以当做不存在，感官上就是说常量是放在内存池中，其他栈中存储的是内存地址。

但实际上不可能，为什么会这么说，因为命名的问题，只是类字段，各个类字段中变量名是可以重复的，而且还有时间上，毕竟还需要再多查一次地址，所以这种不可能。

而线程的运行是很快的，所以对于线程之间共享的变量需要用volatile来处理。保证更改后就立马写回。

于是功能有以下两点。

  • 每次访问变量时候，获取到主内存的最新值
  • 每次修改变量后，立刻回写到住内存。

守护线程

这个该如何理解呢。

首先我们先了解到的线程就是一个一个任务。那么任务代码中肯定可以包括了循环。

在前面中断线程的时候，我们要通过外部结束一个线程是使用interrupt。线程内部是用isInterrupted()来捕获这个通信，这样来结束一个线程。

那么是不是存在死循环线程。简单的例子就是线程内部不对interrupt进行捕获即可。那么这种线程该怎么结束呢。

或许会说为什么要写这种死循环线程，但在多线程中，死循环线程是很常见的。通信，IO。同步线程，或者异步线程。都会用到死循环，因为要一直循环一个代码来进行获取另一个线程是否完成，或者取出什么数据了。唤醒另一个线程什么之类的。

而且这种主线程退出的时候，也就是整个任务结束的时候，是不会手动通知这些循环线程结束的，因为这种循环线程会很多。于是就出现了守护线程。

什么意思，通常我们知道Java程序入口是虚拟机启动main线程。main线程启动其他线程。当所有线程运行结束的时候，虚拟机会退出，进程结束。但是守护线程指的是，不关这个线程有没有结束，只要其他的非守护线程结束了。虚拟机就会退出。

而启动守护线程也很简单。就是添加一个标记即可。

Thread t = new MyThread();
t.setDaemon(true);
t.start();

线程同步

这个问题的出现就在于多个线程同时读写一个内存变量的时候，该怎么办，因为线程的执行是随机的。于是就有了前面线程状态中的等待状态。

毕竟线程新建，运行，终止就完事了。那么出现等待和阻塞就一定是有其他作用，而这些作用就是为了这个，线程之间的同步。

这种方法多了去了，只要能进入到阻塞即可，或者说保证多个为一个单元，即有回滚机制。但这玩意儿有点扯远了。

例子如下

// 多线程
public class Main {
    public static void main(String[] args) throws Exception {
        var add = new AddThread();
        var dec = new DecThread();
        add.start();
        dec.start();
        add.join();
        dec.join();
        System.out.println(Counter.count);
    }
}

class Counter {
    public static int count = 0;
}

class AddThread extends Thread {
    public void run() {
        for (int i=0; i<10000; i++) { Counter.count += 1; }
    }
}

class DecThread extends Thread {
    public void run() {
        for (int i=0; i<10000; i++) { Counter.count -= 1; }
    }
}

答案不可猜测。

为什么，看代码内容，我只要执行的次数是对的，我又不管中间值会是多少。那么就不会出差错。但是问题就在于

n = n + 1；

虽然是一行代码，但是对应的指令是三条。所以这里才是出错的关键点。

两个线程都进入到写回状态了，进行了二次覆盖，那么就意味着其中一个的加减是无效的。那么循环次数就会少一个。

那么处理这个呢。

当然要是闲的蛋疼也可以在第一个线程启动后，在第一个线程内部再启动第二个线程。

但是不好看，也丧失了多线程的意义，所以就有了加锁，或者说，把非原子操作变成原子操作。

死锁

这东西比较难理解点，为什么会这么说，原因就在于，上面锁的内容大概能看懂吧，简单来说就是多个线程需要访问同一个共享变量，或者说是修改同一个变量，问题就在这里，次序不对还好说，反正只要结果对就行，但是因为单次执行的代码块，赋值和取出在指令上是多个的，所以就会出现刚取出，然后其他的结果就放回去了，那么这个时候取出的内容就会过期。自然而然就出现误差。

再往后，一个类创建的对象，如果不加锁，那么就会出现更新不及时，虽然说赋值是原子操作，但是如果有多个语句赋值，那么就会造成读入的数据不成套。然后读取的时候也有可能是不成套的。换算成人话就是，字段相当于共享变量，然后方法就是对这个字段的操作，所以要进行加锁。然后就是一顿巴拉巴拉说加锁应该是加多大的锁，就是这个锁的区间有多大，毕竟加锁是影响效率的。多个字段的赋值读取也是要加锁的。这一个组合拳下来就迷迷糊糊的。

然后说死锁难，就在于是现在是两个锁，因为有嵌套调用，所以会出现卡住的情况。

那么死锁肯定不是指可重入锁，因为当前能获取锁的时候，那么下一个肯定能获取锁。原理也很简单，就是到当前情况的时候，除了重启，否则无法解决。

import java.util.*;

public class Main {
    public static void main(String[] args) throws InterruptedException {
        var q = new TaskQueue();
        var ts = new ArrayList<Thread>();
        for (int i=0; i<5; i++) {
            var t = new Thread() {
                public void run() {
                    // 执行task:
                    while (true) {
                        try {
                            String s = q.getTask();
                            System.out.println("execute task: " + s);
                        } catch (InterruptedException e) {
                            return;
                        }
                    }
                }
            };
            t.start();
            ts.add(t);
        }
        var add = new Thread(() -> {
            for (int i=0; i<10; i++) {
                // 放入task:
                String s = "t-" + Math.random();
                System.out.println("add task: " + s);
                q.addTask(s);
                try { Thread.sleep(100); } catch(InterruptedException e) {}
            }
        });
        add.start();
        add.join();
        Thread.sleep(100);
        for (var t : ts) {
            t.interrupt();
        }
    }
}

class TaskQueue {
    Queue<String> queue = new LinkedList<>();

    public synchronized void addTask(String s) {
        this.queue.add(s);
        this.notifyAll();
    }

    public synchronized String getTask() throws InterruptedException {
        while (queue.isEmpty()) {
            this.wait();
        }
        return queue.remove();
    }
}

这段程序做了什么，首先是添加五个线程启动，然后添加到队列中。这五个线程是获取到任务的线程。

然后再添加一个线程，而这个线程会添加十个队列。用于唤醒。

之后再对移除的队列添加中断。用于线程结束。

另外就是为什么要用while循环而不是if判断，原因在于，线程被唤醒的时候，很有可能没有获取到锁，就已经结束了。那么就跳出了循环，造成错误。

Condition

这些相当于重复内容，至于为什么不能混用，没有看过源码，不知道。

首先一些匹配点是在

synchronized 配合 wait 和notify 来实现线程在条件不满足的时候等待，

后来了解到了reentrantlock，用于代替synchronized。那么wait和notify也桐乡需要代替。

这里选用Condition对象来实现wait和notify功能。

例子如下

class TaskQueue {
    private final Lock lock = new ReentrantLock();
    private final Condition condition = lock.newCondition();
    private Queue<String> queue = new LinkedList<>();

    public void addTask(String s) {
        lock.lock();
        try {
            queue.add(s);
            condition.signalAll();
        } finally {
            lock.unlock();
        }
    }

    public String getTask() {
        lock.lock();
        try {
            while (queue.isEmpty()) {
                condition.await();
            }
            return queue.remove();
        } finally {
            lock.unlock();
        }
    }
}

Condition提供的await()、signal()、signalAll()原理和synchronized锁对象的wait()、notify()、notifyAll()是一致的，并且其行为也是一样的：

  • await()会释放当前锁，进入等待状态；
  • signal()会唤醒某个等待线程；
  • signalAll()会唤醒所有等待线程；
  • 唤醒线程从await()返回后需要重新获得锁。

ReadWriteLock

这个锁的出现，是应对于读写问题，首先我们要明白的是，读写是互斥的，并且写与写之间是互斥的。读与读之间不互斥，这不同于实际资源的请求，放到电路上，读数据，是通过把地址交给访问器，根据地址将数据流流出来，本质上是是并发的，因为可能指针只有那一个，但是执行本来就是顺序执行的，可以做到连续访问，而这个时间段，对于加锁的来说，时间实在是微不足道了。

那么就要想办法进行优化，如果是自己实现，那么就需要两个变量，一个是读写两个锁，但是都需要用到这两个锁，才能完成一次读写，但是在读的时候，可以换成可重入锁。写锁是非重入锁。

那么利用写好的程序，调用吧。

public class Counter {
    private final ReadWriteLock rwlock = new ReentrantReadWriteLock();
    // 注意: 一对读锁和写锁必须从同一个rwlock获取:
    private final Lock rlock = rwlock.readLock();
    private final Lock wlock = rwlock.writeLock();
    private int[] counts = new int[10];

    public void inc(int index) {
        wlock.lock(); // 加写锁
        try {
            counts[index] += 1;
        } finally {
            wlock.unlock(); // 释放写锁
        }
    }

    public int[] get() {
        rlock.lock(); // 加读锁
        try {
            return Arrays.copyOf(counts, counts.length);
        } finally {
            rlock.unlock(); // 释放读锁
        }
    }
}

代码没变，这些都交给底层了。剩下的调用很简单。

stampedlock

乐观锁，这种是用于读写锁的优化，

首先是读写锁是不会出现数据不一致。有一点时间可以去优化，那就是读锁的过程可以允许写锁的加入。

读写锁又被称为悲观锁，因为是假设读的时候写入会发生数据过期。

使用乐观锁需要对读出的数据进行比较，需要验证。

public class Point {
    private final StampedLock stampedLock = new StampedLock();

    private double x;
    private double y;

    public void move(double deltaX, double deltaY) {
        long stamp = stampedLock.writeLock(); // 获取写锁
        try {
            x += deltaX;
            y += deltaY;
        } finally {
            stampedLock.unlockWrite(stamp); // 释放写锁
        }
    }

    public double distanceFromOrigin() {
        long stamp = stampedLock.tryOptimisticRead(); // 获得一个乐观读锁
        // 注意下面两行代码不是原子操作
        // 假设x,y = (100,200)
        double currentX = x;
        // 此处已读取到x=100，但x,y可能被写线程修改为(300,400)
        double currentY = y;
        // 此处已读取到y，如果没有写入，读取是正确的(100,200)
        // 如果有写入，读取是错误的(100,400)
        if (!stampedLock.validate(stamp)) { // 检查乐观读锁后是否有其他写锁发生
            stamp = stampedLock.readLock(); // 获取一个悲观读锁
            try {
                currentX = x;
                currentY = y;
            } finally {
                stampedLock.unlockRead(stamp); // 释放悲观读锁
            }
        }
        return Math.sqrt(currentX * currentX + currentY * currentY);
    }
}

其实最开始的时候也在好奇，怎么检查。这里给出了答案，检查版本号。检测在读锁之后是否写入了写锁，如果发生了写锁，就重新上悲观锁。也就是读锁。但是操作很复杂。

另外这个是不可重入锁。

atomic

还记得我们最开始为什么要加锁么，因为 n = n + 1 不是原子操作，必须加锁，来确保是一个原子操作。但是Java就这一点好，已经有处理好的。就像集合类一样，当我们对一个集合进行增添操作的时候，有一个concurrent类来帮助实现。

我们对这种只需要原子操作的也有一个专门的类。

这种是利用了无锁的方式。原理是CAS，比较和设置。原理就是先取出当前值，设定下一个值，之后进行比较当前值是否是与原先的一致，如果一致，就更新成next。如果不是就一直循环这个操作，直到+1.为什么是不用考虑原先的情况，

因为最开始解决的问题就在于是多个线程同时取值，然后就造成了一些操作减少。所以只需要保证操作次数足够就行了。中间值是什么，不关。

  • 获取当前值：int get()
  • 用CAS方式设置：int compareAndSet(int expect, int update)

这是关于int的方式，至于最后的compareandset则是交给自己来书写CAS的。但是一般情况下我们用封装好的方法即可。

CompletableFuture

这里是针对Future的优化，也就是真正变为异步。

关于异步的定义其实也不复杂，在多个领域都有使用，Python使用最多。

首先异步和串行是在同一概念上。所谓串行是指当要获得线程执行结果的时候，主线程需要设置循环等待。在外表上看，就是只有当下一个任务执行结束才能继续执行，影响多线程进度。

这是针对需要获取结果的多线程而言的。毕竟如果不需要或许结果的话，那么自然是不用管执行顺序。

那么异步是什么，异步是指要获取执行结果的话，那么主线程可以继续执行不用返回结果的程序，当结果执行完毕发送信号，让主线程再次进入处理结果的即可。

这是标准的异步，当然这其中还有很多的中间状态，因为发送信号，主线程还要进入到另一任务中，那么必不可少的是要有个循环，不管是在程序中断时候循环进入还是其他的。这也被称为异步。然后还有互为锚点的，什么乱七八糟的方法都有。

那么现在CompletableFuture是什么呢。

则是跟写一个switch的差不多。省却异步的多种概念，当做普通任务处理即可。

// CompletableFuture
import java.util.concurrent.CompletableFuture;

public class Main {
    public static void main(String[] args) throws Exception {
        // 创建异步执行任务:
        CompletableFuture<Double> cf = CompletableFuture.supplyAsync(Main::fetchPrice);
        // 如果执行成功:
        cf.thenAccept((result) -> {
            System.out.println("price: " + result);
        });
        // 如果执行异常:
        cf.exceptionally((e) -> {
            e.printStackTrace();
            return null;
        });
        // 主线程不要立刻结束，否则CompletableFuture默认使用的线程池会立刻关闭:
        Thread.sleep(200);
    }

    static Double fetchPrice() {
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
        }
        if (Math.random() < 0.3) {
            throw new RuntimeException("fetch price failed!");
        }
        return 5 + Math.random() * 20;
    }
}

其中result是多线程执行结果。

ForkJoin

这个就是原本需要自己手动来分发任务，但是可以用这个，来自动判断一个任务是否足够小，即可以裂变。有点像迭代。但是不会出现栈溢出的情况。

import java.util.Random;
import java.util.concurrent.*;

public class Main {
    public static void main(String[] args) throws Exception {
        // 创建2000个随机数组成的数组:
        long[] array = new long[2000];
        long expectedSum = 0;
        for (int i = 0; i < array.length; i++) {
            array[i] = random();
            expectedSum += array[i];
        }
        System.out.println("Expected sum: " + expectedSum);
        // fork/join:
        ForkJoinTask<Long> task = new SumTask(array, 0, array.length);
        long startTime = System.currentTimeMillis();
        Long result = ForkJoinPool.commonPool().invoke(task);
        long endTime = System.currentTimeMillis();
        System.out.println("Fork/join sum: " + result + " in " + (endTime - startTime) + " ms.");
    }

    static Random random = new Random(0);

    static long random() {
        return random.nextInt(10000);
    }
}

class SumTask extends RecursiveTask<Long> {
    static final int THRESHOLD = 500;
    long[] array;
    int start;
    int end;

    SumTask(long[] array, int start, int end) {
        this.array = array;
        this.start = start;
        this.end = end;
    }

    @Override
    protected Long compute() {
        if (end - start <= THRESHOLD) {
            // 如果任务足够小,直接计算:
            long sum = 0;
            for (int i = start; i < end; i++) {
                sum += this.array[i];
                // 故意放慢计算速度:
                try {
                    Thread.sleep(1);
                } catch (InterruptedException e) {
                }
            }
            return sum;
        }
        // 任务太大,一分为二:
        int middle = (end + start) / 2;
        System.out.println(String.format("split %d~%d ==> %d~%d, %d~%d", start, end, start, middle, middle, end));
        SumTask subtask1 = new SumTask(this.array, start, middle);
        SumTask subtask2 = new SumTask(this.array, middle, end);
        invokeAll(subtask1, subtask2);
        Long subresult1 = subtask1.join();
        Long subresult2 = subtask2.join();
        Long result = subresult1 + subresult2;
        System.out.println("result = " + subresult1 + " + " + subresult2 + " ==> " + result);
        return result;
    }
}

通过这些书写给划分为小的区间。内部通过迭代，来开启新的任务。

可以理解为每次都是运行compute的任务。

看是否在确定范围内，如果不是则就继续新建即可。通过invokeAll来调用。join来获取结果。
