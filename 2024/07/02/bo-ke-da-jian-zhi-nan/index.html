如何访问一个网站

这里以客户端访问服务器部署的网站。其中服务器是架设了以下内容。

前端（HTML、CSS、JavaScript）、网关（NGINX）、后端（Java Spring Boot）和数据库（MySQL）的完整网站

访问流程

  1. 用户输入网址

  2. DNS解析

  3. 建立连接

  4. HTTP请求和响应

  5. 前端渲染

  6. 后端处理

  7. 数据库查询

    以上就是全过程，但是其中还有一个网关，相当于一个看大门的，对进入进出的信息进行管理。

输入网址

客户打开浏览器输入网址

DNS解析

  1. 浏览器缓存，检查本地缓存是否有对应的IP地址
  2. 操作系统缓存，如果浏览器缓存中没有，查询操作系统缓存。
  3. 本地域名服务器，如果操作系统没有，请求配置的DNS服务器
  4. 递归查询。 就是本地域名服务器，根域名服务器，顶级域名服务器，权威域名服务器。依次递归查询。当然还有其他技术，就是查询的时候附带本地域名服务器。当然其他的还是递归查的。
  5. 返回IP地址。 找到IP地之后，依次返回给操作系统，浏览器，并缓存结果。

建立连接

浏览器获取到服务器的IP地址后，与服务器建立连接

  1. TCP三次握手,SYN 浏览器发送SYN请求。
  2. SYN-ACK 服务器回复SYN-ACK响应。
  3. ACK 浏览器发送ACK 请求，连接建立成功

HTTP请求和响应

连接建立欧，浏览器发送HTTP请求到服务器，服务器处理请求并返回响应。

  1. 发送HTTP请求， 浏览器发送 GET/HTTP/1.1 请求到服务器
  2. NGINX接受请求， 根据网关配置将请求转发到后端服务，例如spring boot 应用
  3. Spring Boot 处理请求。接受请求后，调用相关控制器处理请求
  4. 查询数据库。 根据请求内容查询数据库。例如MYSQL 数据库
  5. 生成响应。Spring Boot生成相应。虽然是前端页面，不过现在是json了。然后返回给网关NGINX
  6. NGINX返回响应。NGINX将后端返回的响应发送给浏览器

以上就是一整套流程，但是这里需要补充一点细节，不过只是相比较于只是为了更好的理解流程而已

前端渲染

浏览器接收到服务器返回的HTML相应后，开始解析和渲染页面

  1. 解析HTML。构建DOM树
  2. 解析CSS ，构建CSSOM树
  3. 解析JavaScript。将DOM树和CSSOM树结合，生成渲染树
  4. 布局，计算每个节点的位置和大小
  5. 绘制，将渲染树的每个节点绘制到屏幕上

后端处理

这里以Spring Boot作为后端服务处理，其他的也一样，虽然跟这个应用没啥关系就是了。

处理来自前端的请求，并与数据库交互

  1. 接受请求。controller 接受HTTP请求
  2. 业务逻辑处理，调用service 处理业务逻辑
  3. 数据库操作，服务类调用数据访问层repository与数据库mysql 交互
  4. 生成响应。处理结果并生成相应数据，返回给控制器。
  5. 返回结果，控制器将相应结果返回给客户端。通过网关

数据库查询

Spring Boot应用与MySQL数据库交互，用于获取或者存储数据

  1. 连接数据库，JPA或者JDBC 连接到MySQL数据库，不过mybatis也挺好用的
  2. 执行查询。执行SQL查询用以获取所需服务
  3. 处理结果，处理查询结果，并返回给服务类
  4. 关闭连接，关闭数据库连接释放资源

网关

主要是起代理，负载均衡，安全和缓存的作用

  1. 代理，接受来自浏览器的请求，并将其转发到后端服务
  2. 负载均衡，如果有多个后端服务实例，可以均衡分配请求到不同的实例
  3. 安全，可以配置SSL/TLS加密，访问控制，DDOS保护等安全功能
  4. 缓存，缓存静态内容或者后端响应，减少服务器负载

其中流程是没有改变，只是在前端渲染和后端处理的API接口以及后端返回的数据格式出现了差异。

静态网站和动态网站的区别

静态网站

  1. 内容固定，静态网站的内容都是预先写好的HTML文件，内容不会根据用户的输入或者请求变化
  2. 速度快，因为服务器只需要文件读取和传输
  3. 安全性高，
  4. 易于托管

适用于 个人博客和个人简历，这也是为什么要做这个的原因。虽然最主要的原因是穷吧，

构建和部署工具。 hexo. jekyll/ hugo。

动态网站

  1. 内容动态生成。动态网站的内容可以根据用户的请求实施生成。因为有后端代码进行处理逻辑并生成响应内容
  2. 交互性强。
  3. 以来数据库
  4. 复杂性高

适用于常见的网站。

构建和部署工具，spring Boot。 django express.js

访问流程对比

静态网站

  1. 用户请求，浏览器发送给服务器
  2. 服务器响应，服务器直接返回余先生承德HTML文件
  3. 浏览器渲染，浏览器解析和渲染页面

动态网站

  1. 用户请求，浏览器发送给服务器
  2. 服务器处理，服务器执行后台代码
  3. 动态生成内容，服务器根据处理结果生成HTML内容
  4. 服务器响应，服务器返回生成的HTML内容
  5. 浏览器渲染，浏览器解析和渲染

所以可以看出动态和静态的区别了，主要是在静态网站上无法直接请求查询一些存储的内容，或者说没办法登陆，评论。

当然这些也不是绝对的，因为本身JavaScript也能执行代码在浏览器上，登陆和评论也可以交给其他服务商进行处理。包括这里使用的github action，本质上是静态网站无法做到的，但是通过交给其他服务商也是相当于在静态网站上实现了动态效果。除了要交钱。

安装 Hexo

hexo是一款静态网页生成工具，可以根据设置主题样式和配置文件，生成丰富多彩的网页。

将markdown文件和模板文件转换为静态的HTML文件。主要就是在本地生成静态网站，讲这些静态网站部署到服务器上。

hexo 工作原理

  1. 编写内容，使用markdown 编写内容，放到source/_posts目录下

  3. 生成静态文件，将markdown文件和模板文件转换为静态的HTML文件
  4. 部署，将生成的静态文件部署到Web服务器上

部署到github pages

在前面的部分我们基本上完成本地的代码书写，虽然好像没做什么，一切都是默认的。但对于结果来说我们就已经搭建出一个网站了，现在考虑要怎么部署 到网络上。

因为 前面如果是运行以下命令

hexo clean #清除缓存
hexo new "my test blog"
hexo generate #生成静态页面
hexo server	#运行服务

就会发现文章已经发布了。

所以现在就是要把本地文件部署到网页空间里，例如github pages。

下面会详细写如何创建一个开启pages服务的仓库。

但这前言还需要认真看清。不然可能对一些操作很迷糊。

首先在本地干了什么

除了第一次的初始化。后面再使用就三个命令。

  1. 新建一篇文章
  2. 使用hexo clean先清除以前生成的静态网页，再使用hexo generate 生成新的静态页面
  3. hexo server，开启网络服务

如果说一直 hexo server 运行中，hexo 是会开启文章监控和实时更新，并且是默认实现的，当然也改不了。

那么说也就说一共就两步，创建一篇文章，然后生成新的静态网页（当然因为自动更新的存在，一步就能完事）。

那么就会注意到，我们本身使用的hexo相当于一个框架，把我们写的东西按照配置转换成静态页面，

那么就是说，我们干嘛把这个hexo放到网上，那只是个静态网页，里面只包含HTML，css,JavaScript。这么多乱七八糟的配置没必要上传到网络上。

思路瞬间清晰。

然后现实给一巴掌，或者也不是说给一巴掌吧。那么是怎么只推送生成的静态页面呢，

首先是hexo的工作机制

Hexo 生成的静态页面默认存放在项目目录下的 public 文件夹中。当你运行 hexo generate 或 hexo g 命令时，Hexo 会将所有的源文件（如 Markdown 文章、主题模板等）编译成静态 HTML、CSS 和 JavaScript 文件，并将这些文件输出到 public 文件夹。

具体步骤如下：

  1. 编写内容：你在 Hexo 项目目录的 source 文件夹中编写 Markdown 文章、添加图片和其他资源。
  2. 生成静态页面：运行 hexo generate 命令，Hexo 会读取 source 文件夹中的内容，并使用主题中的模板文件将其编译成静态页面。
  3. 输出到 public 文件夹：生成的静态文件（HTML、CSS、JavaScript 等）会被存放在项目根目录下的 public 文件夹中。

使用hexo generate ，生成静态文件到public 文件夹。hexo server启动本地服务器并试试预览网站，自动生成并监控文件的变化。hexo clean，清理public 文件夹和缓存文件夹，如.deploy_git 和 .deploy_vercel。

所以如果部署到github pages 上，每次想要更新了，就写个文章，然后 hexo generate 其次git add public ,git commit -m “” ，git push ，即可。

这样实现了更新网页的方式。

但是有没有觉得挺麻烦，毕竟人就是会偷懒的，一次让我干这么多的事，这文章是一点都不想写。

那怎么办，

那我们就修改下 ——config.yml。使用个hexo 的部署插件

hexo-deployer-git。

这样做

  1. 修改 _config.yml文件
url: https://mirror18.github.io/
root: /
deploy:
  type: 'git'
  repo: git@github.com:Mirror18/Mirror18.github.io.git
  branch: main

当然root 这里不是必须的，后续会说清楚。

然后就安装hexo的插件。

npm install hexo-deployer-git --save]

这样就省去git add . 等等这一系列的命令。

再次使用就是

hexo gengerate
hexo deploy

就可以推送所有的变更了。

不过关于这个deploy中的branch还是要多说点，首先是这个分支 main代表了什么意思。

仔细想想我们省略了什么，一套流程行云流水，add ,commit ,push。对，关键是在于这个push ,平常使用的时候，push origin master。这是将本地的master分支推送到远程的master分支，因为名字相同，所以省略了冒号加区分。

那现在呢，只是调用了hexo deploy，那么这玩意儿推送的是哪个分支。或者说这玩意儿的分支是怎么规定的，没错，就是这里的main,代表了推送到远程的main分支上。推送的内容就是静态内容，就是public文件夹里面的东西，只有这些，没有源码。如果在运行hexo deploy之前先git push 过，并且用的还不是main分支，那么恭喜你，想办法解决分支合并的问题吧。这玩意儿我到最后是用强制合并才解决的。十分头疼。

那么按理来说这里就可以结束了没见到github action的事啊。别急。实际上还是为了偷懒。

总所周知，人是很懒的，hexo gengerate ，hexo deloy还是太累了，最主要的是我写个这还要其他命令。平常写个git add 怨气就很大了，不行，不想干。

有两种类型的 github pages，一种是使用 用户名.github.io 作为项目名，一种是使用其它名称。虽然看起来只是名字不一样，但两种方式其实是有差异的，前一种方式里，网页静态文件只能存放在 master 分支，所以如果想要把博客源文件也存到同一个仓库，必须使用其它分支来存放，当然也可以使用另一个新的仓库来存放。后一种方式则没这个限制，通常使用名为 gh-pages 作为分支名，Hexo 内默认设置的分支也是叫这个名字。
源文件在master分支，静态文件在gh-pages分支。

配置文件

name: Deploy Hexo to GitHub Pages  # 工作流名称

on:
  push:  # 当推送到指定分支时触发
    branches:
      - main  # 监控的分支名称

jobs:
  build:  # 作业名称
    runs-on: ubuntu-latest  # 运行环境

    steps:
    - name: Checkout repository  # 检出代码库
      uses: actions/checkout@v2  # 使用官方的 checkout action

    - name: Setup Node.js  # 设置 Node.js 环境
      uses: actions/setup-node@v2  # 使用官方的 setup-node action
      with:
        node-version: '14'  # 指定 Node.js 版本

    - name: Cache node modules  # 缓存 node_modules
      uses: actions/cache@v2  # 使用官方的 cache action
      with:
        path: node_modules  # 缓存路径
        key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}  # 缓存的 key
        restore-keys: |
          ${{ runner.os }}-node-  # 恢复的 key 前缀

    - name: Install dependencies  # 安装依赖
      run: npm install

    - name: Generate static files  # 生成静态文件
      run: npx hexo generate

    - name: Deploy to GitHub Pages  # 部署到 GitHub Pages
      env:
        GH_TOKEN: ${{ secrets.GH_TOKEN }}  # GitHub Token，用于身份验证
      run: |
        git config --global user.name 'github-actions[bot]'  # 配置 git 用户名
        git config --global user.email 'github-actions[bot]@users.noreply.github.com'  # 配置 git 邮箱
        npx hexo deploy  # 执行 hexo 部署

上述是一份正规的文档解释。

以下是我的配置

name: 自动部署  #工作流的名称

on: #定义何时出发这个工作流
  push: #当推送到指定分支时候触发工作流
    branches: #指定监控的分支名称
      - master  #当推送到master分支时候出发工作流

  release:  #当发布新版本时候出发工作流
    types:  #发布的类型
      - published #当被发布时候出发工作流

jobs: #作业
  deploy: #定义了一个名为deploy的作业
    runs-on: ubuntu-latest  #指定运行环境为最新的ubuntu
    steps:
    - name: 检查分支
      uses: actions/checkout@v2 #使用的操作，检出代码哭
      with: #提供的参数
        ref: master #指定检出的分支为master

    - name: 安装 Node
      uses: actions/setup-node@v1
      with:
        node-version: "20.13.1"

    - name: 安装 Hexo #执行一下的shell命令，设置时区，全局安装
      run: |
        export TZ='Asia/Shanghai' 
        npm install hexo-cli -g

    - name: 缓存 Hexo
      uses: actions/cache@v1
      id: cache #设置步骤的ID为cache,方便后续的引用
      with: #提供给cache操作的参数。缓存的路径，缓存的key
        path: node_modules
        key: ${{runner.OS}}-${{hashFiles('**/package-lock.json')}}

    - name: 安装依赖
      if: steps.cache.outputs.cache-hit != 'true' #如果缓存没有命中。泽就安装项目依赖
      run: |
        npm install --save

    - name: 生成静态文件
      run: |
        hexo clean
        hexo generate

    - name: 部署 #此处master:master 指从本地的master分支提交到远程仓库的master分支(不是博客的分支写master即可)，若远程仓库没有对应分支则新建一个。如有其他需要，可以根据自己的需求更改。
      run: |
        cd ./public
        git init
        git config --global user.name '${{ secrets.GITHUBUSERNAME }}'
        git config --global user.email '${{ secrets.GITHUBEMAIL }}'
        git add .
        git commit -m "${{ github.event.head_commit.message }} $(date +"%Z %Y-%m-%d %A %H:%M:%S") Updated By Github Actions"
        git push --force --quiet "https://${{ secrets.GITHUBUSERNAME }}:${{ secrets.GITHUBTOKEN }}@github.com/${{ secrets.GITHUBUSERNAME }}/${{ secrets.GITHUBUSERNAME }}.github.io.git" master:master  # GitHub配置
# git commit -m "${{ github.event.head_commit.message }} $(date +"%Z %Y-%m-%d %A %H:%M:%S") Updated By Github Actions"：提交更改，提交信息包括最新的提交信息和当前日期时间。
# git push --force --quiet "https://${{ secrets.GITHUBUSERNAME }}:${{ secrets.GITHUBTOKEN }}@github.com/${{ secrets.GITHUBUSERNAME }}/${{ secrets.GITHUBUSERNAME }}.github.io.git" master:master：强制推送到 GitHub Pages 仓库的 master 分支。

# 所以这里出现的本地master分支是哪个分支。是当前创建好的环境，例如ubuntu环境中这台机器上的master分支
# 所以这里干了什么呢，就是将public里的文件给推送到公开页面中了
