如何创建一个springBoot项目

再此之前希望懂一点，JDBC，Web开发，maven项目管理和构建工具。这些前提。

spring Boot 只是个框架，借用这个东西完成单例注入之类的。至于spring MVC，那则是必须的。

项目定性

首先先要明白我的需求是什么，不管是一个网页，一个API调用。总的来说我们是要为人服务，那么自然就是思考要提供什么服务。

这里以苍穹外卖为例子吧，毕竟时间这么短，我能实际做的项目也不多。

苍穹外卖需要提供一个管理端页面，客户页面。其中管理端页面就是一个web网站，客户使用的是微信小程序。

那么我们采用编写的spring Boot 就需要提供两套的controller，调用的API接口不一样，而且所提供的服务也有权利限制。

因为一个真实的项目很大，包括前后端。所以重点就是后端开发，中间传递的是json信息。后端从前端拿到的就是URL，并且每个URL需要提供什么样的数据。这就是我们后端所需要考虑的，至于前端拿到数据怎么展示就不是我们的问题了。

确定URL

也就是确定都有哪些URL，需要提供哪些信息，并且编写是怎么编写的。

确定使用的框架

毕竟只是需要根据URL的请求，我们只是做一个响应内容的填充。django，mvc，都可以提供。

所以这里是用spring MVC 来进行提供。（实际上用的也只是servelet).

确定好后端用什么开展，那么就启动吧。

但还是会说，草，好累，我创建一个spring框架还要写这么多的配置，还要写一个xml来专门讲述这种依赖关系。不想干了，而且还那么容易出错。

于是就出现了用注解代替配置文件，也就是xml。

于是我们就这样做。需要交给spring管理的bean，也就是UserService就在上面添加一个@Component，需要注入的对象，也就是MailService mailServie添加@Autowired。Main类，也就是启动类，现在改名为AppConfig的类上面添加上@Configuration 和 @ComponentScan这两个注解.

最后的成品就是这样

这里讲解下@Configuration，表明这是一个配置类，不参与bean的管理，但是参与如何创建一个入口。

@ComponentScan这是表明，自动搜索当前嘞所在的包和子包，把所有标注为@Component的Bean自动创建出来，根据@Autowired进行装配。

但这里提前注意的是，虽然我们不用xml来配置启动，同样的，我们也失去了自定义导入信息。例如上面写配置文件的时候，创建datasource的时候，我们不仅写明了他需要那个类才能创建这个对象，还有字段的赋值，name,jdbc这些字段的赋值。

如果前面注解看过的话，就明白所谓的注解就是加了一段信息。这跟Python的注解不一样，Python的注解是相当于包裹了一段程序。当然Java的注解也可以是说包裹程序，就看反射那是怎么处理的了。我添加注解只是让spring识别到这个是需要实例化，也就是交给spring处理，但是引进的jar上面可是没添加注解的，而且同样的注解也没有字段用来包裹name,jdbc这些字段信息。所以还是要写配置文件，但是不用写各个class的依赖关系。

同样的，我们不用配置信息配置，那么还少了一个周期，在设计模式中我们学过单例和原型。这是两种创建方式，虽然没啥大差别，但是用复制的时候还是差别很大的。所以我们也需要在创建对象的时候注明这是单例还是原型。

那怎么标注呢，而且还有第三方的bean也需要自动注入，而且针对接口相同不同实现类的bean，我们是要一起处理的，该怎么办呢，注解也罢，xml配置信息也是要有的。

因为使用注解，所以方法就灵活很多，需要什么自己使用方法装填就好。

但同时需要注意的点是这里，因为这里是面对对象，所以调用的来的就是一个对象，虽然我们在说，spring 控制了我们对象的创建和销毁，但是在创建对象的时候是根据构造器来的，虽然说也有方法创建对象，但我们需要注意的是，在本次对象创建的时候，一般会有一个初始化的操作，还有在销毁的时候需要一个销毁的操作。

其实这里不知道要不要写，因为除了销毁的事，我们初始化的时候可以手动初始化，销毁没办法，销毁是随着进程的结束而销毁，再加上我们创建的都是单例，手动销毁会出问题，因为有多线程的存在。当然要说线程同步，加锁，这倒是没啥。

不过为了简化，也就是为了偷懒，所以在一个bean中，我们可以做以下操作。

@Component
public class MailService {
    @Autowired(required = false)
    ZoneId zoneId = ZoneId.systemDefault();

    @PostConstruct
    public void init() {
        System.out.println("Init mail service with zoneId = " + this.zoneId);
    }

    @PreDestroy
    public void shutdown() {
        System.out.println("Shutdown mail service");
    }
}

这些名字无所谓，因为spring 只根据annotation查找无参数方法，对方法名不做要求。

至于说指定别名，则是@Bean(“name”)或者@Bean+@Qualifier(“name”)。使用的时候就是@Autowired+@Qualifier(“name”)。当然要是偷懒，则在创建的时候则对默认的加上一个@Primary。即@Bean+@Primary+@Qualifier(“name”).

按照编程语言的尿性，任何工具都会有两种写法。连注入都有两种呢，除了@Autowired还有一个@Resource。对于普通编写的人来说差不哪去。

那么创建第三方bean的时候，除了在配置类中@Bean进行装填，还可以用工厂模式进行创建

@Component
public class ZoneIdFactoryBean implements FactoryBean<ZoneId> {

    String zone = "Z";

    @Override
    public ZoneId getObject() throws Exception {
        return ZoneId.of(zone);
    }

    @Override
    public Class<?> getObjectType() {
        return ZoneId.class;
    }
}

这里是一定要调用factorybean这个接口，然后用getobject()创建真正的bean.用getobjecttype()则是返回了bean的类型。为什么会有这样的呢，在配置类中使用@Bean创建的时候我们也经常是用不同于实际创建的类进行创建，返回类型是自定义的，所以就区别在这里。至于说这个用的多少，那纯纯没用过。就那样说，都有两种方法，那种方法写的代码少，不用太过费心思，就用那个。

所以综上所述，我们要创建一个spring，则就是@Configuration + @ComponentScan在AppConfig类上。自定义的组件就用@Component，三方bean则就用@Bean。装配bean 则就用@Reource(name)来代替，当然这个name 也可以换成type,不过就是@Resource(type=”name”)这样代替@Autowired。来进行一个spring的管理。

拓展代码

什么是动态代理，就是说给一个接口。通过一个接口，通过反射去实现一个接口的实现类。

注意到动态代理本质上就是通过一个接口通过反射创建出一个实现类。至于说是增强业务，那么在通过反射实现一个实现类的时候。我传入需要增强的实现类对象就行。

这也就是说静态代理和动态代理区别不大的原因。因为静态代理是我自己使用proxy模式，自己创建一个新的实现类就完事。动态代理则就是只要属于同一个接口，都可以把实现类给增强，少了很多的代码。这也是动态代理的本质，主要减少代码量。

所以注意到动态代理，就是通过反射实现接口，并且在这其中我自己本人没有创建一个新的实现类。那么对于spring框架怎么办呢，在spring下实现动态代理怎么办，毕竟那玩意儿写的是真复杂。

目的就是在原有实现的功能上，不去修改代码，而是新增代码就可以实现对功能的拓展。

而且对于动态代理来说，虽然是添加了动态代理，但是在调用的时候也是需要新创建一个动态代理对象，将要代理的实现类给传入进去。对于spring框架来说，都是自动装配的，再加上一个包裹类，这对象都不是我创建的，我怎么加。

于是就有了aop技术。

技术是为人服务的，再怎么好，也要会调用，那么怎么调用。

最简单的方法，只需要多创建一个bean就行。

@Aspect
@Component
public class LoggingAspect {
    // 在执行UserService的每个方法前执行:
    @Before("execution(public * com.itranswarp.learnjava.service.UserService.*(..))")
    public void doAccessCheck() {
        System.err.println("[Before] do access check...");
    }

    // 在执行MailService的每个方法前后执行:
    @Around("execution(public * com.itranswarp.learnjava.service.MailService.*(..))")
    public Object doLogging(ProceedingJoinPoint pjp) throws Throwable {
        System.err.println("[Around] start " + pjp.getSignature());
        Object retVal = pjp.proceed();
        System.err.println("[Around] done " + pjp.getSignature());
        return retVal;
    }
}

@Configuration
@ComponentScan
@EnableAspectJAutoProxy
public class AppConfig {
    ...
}

这样就完成了对一个类的所有public 方法进行添加代理的方法。

当然还有其他周期。

@Before：这种拦截器先执行拦截代码，再执行目标代码。如果拦截器抛异常，那么目标代码就不执行了；

@After：这种拦截器先执行目标代码，再执行拦截器代码。无论目标代码是否抛异常，拦截器代码都会执行；

@AfterReturning：和@After不同的是，只有当目标代码正常返回时，才执行拦截器代码；

@AfterThrowing：和@After不同的是，只有当目标代码抛出了异常时，才执行拦截器代码；

@Around：能完全控制目标代码是否执行，并可以在执行前后、抛异常后执行任意拦截代码，可以说是包含了上面所有功能。

  2. 编写Bean时，如果可能会被代理，就不要编写public final方法。

一共有下面的级别，

默认的事务传播级别是REQUIRED，它满足绝大部分的需求。还有一些其他的传播级别：

SUPPORTS：表示如果有事务，就加入到当前事务，如果没有，那也不开启事务执行。这种传播级别可用于查询方法，因为SELECT语句既可以在事务内执行，也可以不需要事务；

MANDATORY：表示必须要存在当前事务并加入执行，否则将抛出异常。这种传播级别可用于核心更新逻辑，比如用户余额变更，它总是被其他事务方法调用，不能直接由非事务方法调用；

REQUIRES_NEW：表示不管当前有没有事务，都必须开启一个新的事务执行。如果当前已经有事务，那么当前事务会挂起，等新事务完成后，再恢复执行；

NOT_SUPPORTED：表示不支持事务，如果当前有事务，那么当前事务会挂起，等这个方法执行完成后，再恢复执行；

NEVER：和NOT_SUPPORTED相比，它不但不支持事务，而且在监测到当前有事务时，会抛出异常拒绝执行；

NESTED：表示如果当前有事务，则开启一个嵌套级别事务，如果当前没有事务，则开启一个新事务。

上面这么多种事务的传播级别，其实默认的REQUIRED已经满足绝大部分需求，SUPPORTS和REQUIRES_NEW在少数情况下会用到，其他基本不会用到，因为把事务搞得越复杂，不仅逻辑跟着复杂，而且速度也会越慢。

定义事务的传播级别也是写在@Transactional注解里的：

@Transactional(propagation = Propagation.REQUIRES_NEW)
public Product createProduct() {
    ...
}

或许会吐槽事务。这玩意儿是怎么确定事务传播级别的。虽然是说用ThreadLocal来保证传播级别。同样的也证明了一件事。事务只能在一个线程内，如果开启两个现成，那就是两个事务。

public abstract class AbstractDao<T> extends JdbcDaoSupport {
    private String table;
    private Class<T> entityClass;
    private RowMapper<T> rowMapper;

    public AbstractDao() {
        // 获取当前类型的泛型类型:
        this.entityClass = getParameterizedType();
        this.table = this.entityClass.getSimpleName().toLowerCase() + "s";
        this.rowMapper = new BeanPropertyRowMapper<>(entityClass);
    }

    public T getById(long id) {
        return getJdbcTemplate().queryForObject("SELECT * FROM " + table + " WHERE id = ?", this.rowMapper, id);
    }

    public List<T> getAll(int pageIndex) {
        int limit = 100;
        int offset = limit * (pageIndex - 1);
        return getJdbcTemplate().query("SELECT * FROM " + table + " LIMIT ? OFFSET ?",
                new Object[] { limit, offset },
                this.rowMapper);
    }

    public void deleteById(long id) {
        getJdbcTemplate().update("DELETE FROM " + table + " WHERE id = ?", id);
    }
}

    }
    return false;
}
    
    //Update
    public void updateUser(Long id, String name) {
    User user = sessionFactory.getCurrentSession().byId(User.class).load(id);
    user.setName(name);
    sessionFactory.getCurrentSession().merge(user);
}
}

JPA的使用

@Component
@Transactional
public class UserService {
    //可以更安全的被多线程安全的共享
    @PersistenceContext
    EntityManager em;
    //查询
    public User getUserById(long id) {
    User user = this.em.find(User.class, id);
    if (user == null) {
        throw new RuntimeException("User not found by id: " + id);
    }
    return user;
}
    public User fetchUserByEmail(String email) {
    // JPQL查询:
    TypedQuery<User> query = em.createQuery("SELECT u FROM User u WHERE u.email = :e", User.class);
    query.setParameter("e", email);
    List<User> list = query.getResultList();
    if (list.isEmpty()) {
        return null;
    }
    return list.get(0);
}
}

public interface UserMapper {

    @Select("SELECT * FROM users LIMIT #{offset}, #{maxResults}")
List<User> getAll(@Param("offset") int offset, @Param("maxResults") int maxResults);
    
    //插入
    @Options(useGeneratedKeys = true, keyProperty = "id", keyColumn = "id")
@Insert("INSERT INTO users (email, password, name, createdAt) VALUES (#{user.email}, #{user.password}, #{user.name}, #{user.createdAt})")
void insert(@Param("user") User user);

    @Update("UPDATE users SET name = #{user.name}, createdAt = #{user.createdAt} WHERE id = #{user.id}")
void update(@Param("user") User user);

@Delete("DELETE FROM users WHERE id = #{id}")

}

web开发

为什么单独拎出来，我们配置好框架，链接好数据库，拓展原有代码之后，是要提供服务的。

可以很前线的说这里提供的就是一个网页服务，也可以说这是提供一个API服务。根据请求的URL。我们只需要传递回去一个JSON数据即可。或者说传递回去一个页面。

对于Java来说，第一款网络服务是什么，我想应该是servlet 。通过用Tomcat调用war包，放到服务器上，成功的运行起来。

那么对于servlet怎么操作呢，很简单。

首先要明白怎么建立链接。首先资源是部署到服务器上，服务器对外提供可供链接的request和response。一个是请求，一个是响应。通过请求获取到请求的路径和要请求的资源。响应则就是要返回的内容，要返回内容和内容类型。

所以有了以上的条件，是怎么处理request和response的。我们是通过TCP或者UDP的协议，按照HTTP协议进行连接。所以如果按照应用层，表示层，绘画层，传输层，网络层，链路层，物理层来书写一份链接的话，就忒麻烦。

就算用TCCP/IP模型来说，只有四层，但是也麻烦，首先是在应用层确定要使用的端口发送和接受消息，对消息进行加密解密，维持会话。通过传输层确定是以那种协议保证正常连接。IP层提供前往的地址，网络接口层则是告诉电脑应该走那个网关，或者说怎么传递。如果细说的话，HTTP是建立在TCP协议之上。TCP使用IP层提供的服务。

如果说从头设计一份完整的连接则过于麻烦，因为涉及到底层的调用，这需要多种事务进行交互。所以我们就进行了封装。

一个应用程序通过一个socket简历远程连接，socket内部通过TCP/IP协议把数据传输到网络。对于Java来说只提供了socket相关的类就封装了操作系统提供的接口。socket是什么，就是IP+端口。

对于服务器也就是资源提供者，socket就是制定的IP地址和指定的端口号

对于客户端也就是资源请求者，socket就是他所在的计算机的IP地址和由操作系统分配的随机端口号。

对于服务器端

public class Server {
    public static void main(String[] args) throws IOException {
        ServerSocket ss = new ServerSocket(6666); // 监听指定端口
        System.out.println("server is running...");
        for (;;) {
            Socket sock = ss.accept();
            System.out.println("connected from " + sock.getRemoteSocketAddress());
            Thread t = new Handler(sock);
            t.start();
        }
    }
}

class Handler extends Thread {
    Socket sock;

    public Handler(Socket sock) {
        this.sock = sock;
    }

    @Override
    public void run() {
        try (InputStream input = this.sock.getInputStream()) {
            try (OutputStream output = this.sock.getOutputStream()) {
                handle(input, output);
            }
        } catch (Exception e) {
            try {
                this.sock.close();
            } catch (IOException ioe) {
            }
            System.out.println("client disconnected.");
        }
    }

    private void handle(InputStream input, OutputStream output) throws IOException {
        var writer = new BufferedWriter(new OutputStreamWriter(output, StandardCharsets.UTF_8));
        var reader = new BufferedReader(new InputStreamReader(input, StandardCharsets.UTF_8));
        writer.write("hello\n");
        writer.flush();
        for (;;) {
            String s = reader.readLine();
            if (s.equals("bye")) {
                writer.write("bye\n");
                writer.flush();
                break;
            }
            writer.write("ok: " + s + "\n");
            writer.flush();
        }
    }
}

咱慢慢 看内容。首先是通过new ServerSocket创建一个端口号，也就是如果有请求通过这个端口进行链接的话。再成功有程序链接进来，也就是ss.accept()成功。则就对这个socket实例进行处理此次请求，看内容就是创建了一个线程。至于说包裹了两层，则是最外层是多线程处理，最内层才是对socket的处理。

通过socket获取到输入和输出流。通过inputstream获取到请求内容，再通过outputstream输出内容。也就是request和response。相当于当做一个文件进行调用了。

而客户端，也就是如何发送一个请求

public class Client {
    public static void main(String[] args) throws IOException {
        Socket sock = new Socket("localhost", 6666); // 连接指定服务器和端口
        try (InputStream input = sock.getInputStream()) {
            try (OutputStream output = sock.getOutputStream()) {
                handle(input, output);
            }
        }
        sock.close();
        System.out.println("disconnected.");
    }

    private static void handle(InputStream input, OutputStream output) throws IOException {
        var writer = new BufferedWriter(new OutputStreamWriter(output, StandardCharsets.UTF_8));
        var reader = new BufferedReader(new InputStreamReader(input, StandardCharsets.UTF_8));
        Scanner scanner = new Scanner(System.in);
        System.out.println("[server] " + reader.readLine());
        for (;;) {
            System.out.print(">>> "); // 打印提示
            String s = scanner.nextLine(); // 读取一行输入
            writer.write(s);
            writer.newLine();
            writer.flush();
            String resp = reader.readLine();
            System.out.println("<<< " + resp);
            if (resp.equals("bye")) {
                break;
            }
        }
    }
}

客户端则就没那么多将就，没有在其中指定自己的端口号。而是置顶了请求地址和请求端口号。

通过获取到inputstream和outputstream。写入请求内容和等待响应内容，然后进行处理。至于包裹那么多层，对吼成了bufferedwriter,则是Java的IO支持的，通过包裹来进行添加功能。也就是相当于当做一个文件处理。

当然这是TCP连接，如果换成UDP连接，则是因为没有流的概念，或者说没有持久连接的概念，无法当做一个文件进行读写。只能当做一个字段读写。关于UDP的编写就不做多叙述，只是同一套东西的不同使用。虽然代码不同，但逻辑相同。
